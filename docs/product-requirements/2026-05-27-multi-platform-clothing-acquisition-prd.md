# 多平台服装 AI 获客工作台产品需求文档

## 1. 背景

AitoBee 当前已经具备 AI 内容生成、多平台发布、小红书插件/Bridge、作品数据抓取、评论检索、账号管理和任务市场等基础能力。客户侧需求聚焦于服装品类在小红书、抖音、快手三个平台上的获客转化，希望通过 AI 生成内容、自动/半自动互动、话术迭代和数据归因，提升从内容曝光到微信线索的转化效率。

`agent_demo` 仅作为功能模块参考，不直接套用实现。可参考的模块包括数据看板、内容管理、引流管理、账号管理；正式产品应基于 AitoBee 现有技术栈、账号体系、插件能力和数据模型建设。

目标演示节点为 2026-06-09 前形成可演示版本。第一期应优先完成“可演示、可解释、可继续运营”的获客闭环，不承诺完整自动成交归因和全自动私信机器人。

## 2. 产品定位

产品名称：多平台服装 AI 获客工作台。

产品定位：面向服装商家/运营团队的小红书、抖音、快手内容获客工具，帮助运营人员完成多账号内容生成、发布排期、评论/私信线索管理、话术迭代和效果追踪。

核心价值：

1. 降低服装商家多平台内容生产成本。
2. 将评论、私信、微信引导等获客动作结构化管理。
3. 用数据反馈优化内容模板、引流钩子和回复话术。
4. 为后续从单客户项目沉淀为可复制 SaaS 模块打基础。

## 3. 用户角色

### 3.1 商家老板

关注结果：新增线索、加微信人数、内容曝光、投入产出、是否能卖货。

典型诉求：

- 快速看到每天发了什么、带来了多少互动和线索。
- 看到哪类内容、哪套话术、哪个账号效果更好。
- 不希望系统过度复杂，需要运营能直接使用。

### 3.2 运营人员

关注效率：素材管理、AI 生成、发布排期、评论回复、线索跟进。

典型诉求：

- 输入产品图、款式、价格、尺码和卖点后快速生成适配小红书、抖音、快手的平台内容。
- 能审核、修改、排期和发布内容。
- 能集中查看评论和意向用户，并获取建议回复。
- 能手动标记是否已私信、是否已引导微信、是否已加微信。

### 3.3 平台管理员/交付团队

关注交付：配置客户账号、配置话术、查看运行状态、排查插件和数据问题。

典型诉求：

- 快速初始化服装行业模板和演示数据。
- 能查看小红书、抖音、快手账号和插件/Bridge 是否可用。
- 能定位内容生成、发布、评论抓取、线索统计的问题。

## 4. 目标与非目标

### 4.1 第一期目标

1. 建立一个“多平台获客工作台”入口。
2. 支持服装产品内容生成、审核、排期和发布流转。
3. 第一期支持小红书、抖音、快手三个平台。
4. 支持多账号统一管理、账号筛选、账号级运营配置和账号状态展示。
5. 支持三平台作品数据和真实评论快照抓取，按平台能力分级展示。
6. 支持引流钩子和回复话术配置。
7. 支持评论/私信线索的集中管理和人工状态标记。
8. 支持曝光、互动、评论、意向、引导微信、加微信的漏斗看板。
9. 支持基于效果数据的内容策略和话术迭代建议。

### 4.2 第一期非目标

1. 不做完整 CRM，不管理微信内的后续销售过程。
2. 不承诺自动识别真实成交金额和退货率。
3. 不承诺全自动私信机器人。
4. 不重做 AitoBee 现有账号体系和发布体系。
5. 不直接合并 `agent_demo` 的前后端实现。
6. 不在第一期支持除小红书、抖音、快手以外的社媒平台。
7. 不在第一期统计成交金额、订单金额、退货率和佣金分成，只统计到“加微信线索”。

## 6. 功能需求

### 6.1 数据看板

目标：让商家和运营快速理解当前小红书、抖音、快手获客效果。

#### 6.1.1 指标卡片

P0 指标：

- 今日发布数
- 今日曝光/预估阅读
- 今日互动数
- 今日评论数
- 今日意向线索数
- 今日引导微信数
- 今日加微信数
- 7 日整体转化率
- 平台分布
- 活跃账号数

转化率口径：

- 曝光到评论率 = 评论数 / 曝光数
- 评论到意向率 = 意向线索数 / 评论数
- 意向到引导微信率 = 引导微信数 / 意向线索数
- 引导到加微信率 = 加微信数 / 引导微信数
- 总获客率 = 加微信数 / 曝光数

#### 6.1.2 趋势图

P0 支持 7 日趋势：

- 曝光
- 点赞
- 收藏
- 评论
- 意向线索
- 加微信

P1 支持 30 日趋势、平台对比和账号对比。

#### 6.1.3 作品排行

P0 展示作品列表排行：

- 作品标题/封面
- 平台
- 发布账号
- 发布时间
- 曝光/预估阅读
- 点赞、收藏、评论、分享
- 意向线索数
- 加微信数
- 使用的话术策略/内容模板

P1 支持按平台、账号、时间、内容风格、引流钩子筛选。

### 6.2 内容管理

目标：把服装产品素材转成小红书、抖音、快手可发布内容，并形成审核和发布闭环。

#### 6.2.1 AI 生成内容

P0 输入项：

- 发布账号
- 发布平台，支持小红书、抖音、快手单选或多选
- 产品图片或参考图片
- 产品名称
- 产品类型，如裙子、牛仔裤、通勤套装、针织衫
- 价格区间
- 尺码范围
- 面料/版型/卖点
- 内容风格，如甜美风、通勤风、韩系、显瘦、小个子、度假风
- 是否自动附加引流钩子

P0 输出项：

- 标题
- 正文
- 话题标签
- 平台适配版本，包括小红书笔记文案、抖音短视频标题/描述、快手短视频标题/描述
- 建议发布时间
- 推荐引流钩子
- 内容策略说明，例如主打“显瘦”“百元内”“场景穿搭”

P1 输出项：

- 多版本标题
- 多版本正文
- 评论区预埋问题建议
- 图片生成或视频生成建议

#### 6.2.2 内容状态流转

P0 状态：

- 草稿
- 待审核
- 已通过
- 已排期
- 已发布
- 发布失败

P0 操作：

- 预览
- 编辑
- 通过审核
- 退回修改
- 设置发布时间
- 进入发布流程
- 删除

#### 6.2.3 与现有能力的关系

- AI 视频/图文生成复用当前 `ai/draft-generation` 能力。
- 发布复用当前 `PublishDialog` 和现有插件/Bridge 发布能力。
- 素材复用现有素材/资产能力。
- 不新增独立发布通道。

### 6.3 引流管理

目标：把“钩子、话术、转化路径”变成可配置、可复盘、可迭代的策略资产。

### 6.7 平台评论能力与接入边界

目标：明确“已登录开放平台账号”和“具备评论读取/回复能力”不是同一件事，避免演示和实施阶段误判能力边界。

#### 6.7.1 小红书评论能力

平台能力判断：

- 当前项目小红书评论能力主要依赖浏览器插件、本地 XHS Bridge 和同一浏览器内的小红书登录态。
- 当前可用能力不是已确认的小红书官方 OpenAPI 直连能力，不应在产品或交付文档中描述为“小红书开放平台评论接口”。
- 真实评论读取可以通过老插件请求小红书 Web 评论接口，或通过 Bridge 打开小红书作品页并从页面状态中提取作品详情和评论快照。
- 评论回复可以通过插件自动化或小红书 Web 评论请求完成，但第一期应默认人工确认后执行。

当前项目状态：

- 前端已有小红书一级评论列表、二级评论列表的数据转换和请求封装。
- 老插件可请求一级评论接口 `/api/sns/web/v2/comment/page` 和二级评论接口 `/api/sns/web/v2/comment/sub/page`。
- 无老插件时，可通过 XHS Bridge 打开作品页并捕获页面中的作品详情和评论快照。
- 一级评论发布优先使用 `unifiedInteraction` 自动化方案。
- 二级评论回复或老插件回退路径可请求 `/api/sns/web/v1/comment/post`，并通过 `target_comment_id` 指定被回复评论。
- 平台管理器当前标注“小红书不支持私信”，因此小红书第一期转化动作应以公开评论回复、人工状态标记和后续线下/其他渠道跟进为主。

一期实现要求：

- 在多账号管理中展示小红书运行状态：本地后端是否启动、Bridge 是否连接、Chrome 扩展是否加载、小红书网页是否登录、账号是否匹配。
- 将评论抓取结果统一保存为后端快照，记录作品 ID、xsec_token、评论 ID、用户信息、评论内容、分页游标、抓取时间和数据来源。
- 补齐评论分页、游标、重复评论去重、失败重试和抓取批次记录。
- 回复前必须经过人工确认；回复结果需回写后端，记录执行账号、回复目标、回复内容、成功/失败、失败原因、平台原始返回和截图。
- 增加账号级回复频控：每日回复上限、最小回复间隔、失败次数暂停、敏感词/联系方式拦截。
- 公开评论回复不得直接包含微信号、企微号、手机号等联系方式。
- Bridge 路径如果要完整获取二级评论，需要补充展开“查看更多回复”、滚动加载和分页抓取能力；否则只能作为页面可见评论快照。

风险与兜底：

- 小红书网页结构、登录态、风控策略变化会影响插件/Bridge 稳定性。
- V0 演示需保留“插件/Bridge 未就绪”的配置引导和演示种子数据兜底。
- UI 必须标记数据来源为插件接口、Bridge 页面捕获、人工快照或演示种子数据。

#### 6.7.2 抖音评论能力

平台能力判断：

- 抖音开放平台提供视频评论相关 OpenAPI，包括评论列表、评论回复列表、回复视频评论。
- 相关能力需要应用申请 `item.comment` 权限，并由用户授权对应 scope。
- 回复视频评论只能回复授权用户自己发布的视频。
- 评论列表和评论回复列表仅适用于存在且公开的视频，私密、删除、仅好友可见等状态可能返回参数错误或无数据。

当前项目状态：

- 当前后端抖音 OAuth scope 仅包含 `user_info,data.external.user`，未包含 `item.comment`。
- 当前 `DouyinApiService` 只实现授权、用户信息、视频列表、client token、ticket/share schema 等能力，未实现评论列表、评论回复列表、评论回复接口。
- 当前统一互动服务 `EngagementService` 未注册抖音 provider。
- 当前前端抖音评论能力主要是插件自动化发一级评论；评论列表和子评论列表仍是 TODO，不是开放平台直连能力。

一期实现要求：

- 抖音如果走官方 OpenAPI，需要先申请并开通 `item.comment`。
- 修改授权 scope 后，已有抖音账号需要重新授权。
- 后端新增抖音评论 API 封装：评论列表、评论回复列表、回复视频评论。
- 新增抖音 engagement provider，接入统一评论拉取、建议回复、人工确认后回复流程。
- 前端在账号状态中展示“评论权限未开通/需重新授权/可用/失败”。

#### 6.7.3 快手评论能力

平台能力判断：

- 当前可确认的项目接入范围是快手账号授权、用户公开信息、视频上传发布、视频列表、视频详情和删除。
- 当前项目使用的快手 scope 为 `user_info,user_video_publish,user_video_info`，不包含明确的评论读取或回复权限。
- 现阶段未能从公开可检索的快手文档中确认与抖音 `item.comment` 等价的评论列表/评论回复 OpenAPI；快手 OpenAPI 目录需在官方控制台或 JS 文档页进一步确认。

当前项目状态：

- 当前 `KwaiApiService` 未实现评论列表或评论回复接口。
- 当前 `KwaiService` 未提供评论读取、评论回复或 engagement provider。
- 前端没有快手平台评论插件模块。

一期实现要求：

- 快手不默认承诺开放平台直连评论读取/回复。
- 需要客户或交付团队在快手开放平台后台确认是否可申请评论读取/回复权限及接口文档。
- 若官方能力可用，再补充 scope、账号重新授权、后端 API 封装和统一 engagement provider。
- 若官方能力不可用或无法按期申请，V0 演示使用“作品数据/评论抓取能力待配置”状态、人工录入快照或插件/Bridge 方案兜底，但必须在 UI 中明确标记数据来源。

#### 6.7.4 产品展示规则

P0 展示：

- 账号级评论能力状态：未配置、待授权、权限不足、可抓取、抓取失败、需人工处理。
- 作品级评论数据来源：官方 OpenAPI、插件/Bridge、人工快照、演示种子数据。
- 抓取失败原因：权限未开通、账号需重新授权、作品非公开、平台风控、插件未连接、接口异常。

P0 约束：

- 小红书数据来源默认按插件/Bridge 真实网页会话能力处理，不按官方 OpenAPI 直连能力承诺。
- “登录开放平台”只代表具备应用配置或账号授权入口，不代表已具备评论读取/回复权限。
- 公开评论回复仍需人工确认后执行，不允许自动公开发送包含微信号的话术。
- 快手评论能力在官方权限确认前不得作为确定性交付承诺。

## 7. 关键用户流程

### 7.1 内容发布获客流程

1. 运营进入多平台获客工作台。
2. 选择发布平台和一个或多个账号。
3. 上传产品图并填写款式、价格、尺码、卖点。
4. 系统生成标题、正文、话题和引流钩子。
5. 运营审核并修改内容。
6. 运营设置发布时间或立即发布。
7. 系统调用现有发布能力完成发布。
8. 发布后作品进入效果追踪列表。

### 7.2 评论线索跟进流程

1. 运营输入小红书、抖音或快手作品链接，或从已发布作品进入数据抓取。
2. 系统抓取作品互动数据和评论。
3. 系统识别潜在线索评论，如问价格、求链接、问尺码。
4. 系统生成建议回复。
5. 运营人工确认后回复或私信。
6. 运营将线索状态标记为已回复、已私信、已引导微信或已加微信。
7. 数据看板自动更新漏斗指标。

### 7.3 策略迭代流程

1. 系统按平台、作品、账号、钩子和话术统计转化效果。
2. 运营查看表现较好的内容模板和话术。
3. 运营调整钩子权重或停用低效话术。
4. 新生成内容优先使用高效策略。
5. 继续追踪下一批内容效果。

## 8. 数据口径

### 8.1 基础指标

- 曝光/阅读：优先使用平台实际可抓取数据；若不可用，则以互动量估算，并在 UI 标明“预估”。
- 互动数：点赞 + 收藏 + 评论 + 分享。
- 评论数：作品评论数量。
- 意向线索数：被系统或人工标记为有购买/咨询意向的评论或私信。
- 引导微信数：运营使用微信引导话术或人工标记已引导微信的线索数。
- 加微信数：运营人工标记已加微信的线索数。

### 8.2 归因口径

第一期采用弱归因：

- 线索归因到来源作品。
- 作品归因到平台、发布账号、内容模板、引流钩子。
- 加微信归因到线索来源作品和最后一次使用的话术模板。

第二期再考虑强归因：

- 微信二维码/短链唯一化。
- 用户评论 ID 与私信用户匹配。
- 销售订单导入或 CRM 对接。

## 11. 验收标准

## 14. 已确认结论与待确认问题

### 14.1 已确认结论

1. 第一阶段支持小红书、抖音、快手三个平台，演示优先级为小红书优先，其次抖音、快手。
2. 三个平台第一期都需要获取真实评论，不采用纯手动导入评论替代。
3. 客户自行配置并维护自己的平台账号。
4. 公开评论话术不允许直接出现微信号；微信号只在后续私聊/人工确认场景中给出。
5. 第一阶段需要支持多运营人员分配线索。
6. 第一阶段只统计到“加微信线索”，不统计成交金额。
7. 小红书当前已有插件/Bridge 路径可获取真实评论并执行评论回复，但不是已确认的官方 OpenAPI 直连能力。
8. 小红书第一期需要补齐账号/插件/Bridge 状态、评论快照持久化、分页去重、回复回写、频控和失败原因展示。
9. 抖音开放平台具备评论列表、评论回复列表和回复视频评论能力，但需要申请 `item.comment` 权限，并要求账号重新授权对应 scope。
10. 当前项目尚未实现抖音官方评论读取/回复接口，也未在统一互动服务中注册抖音 provider。
11. 当前项目尚未实现快手评论读取/回复接口，也未确认快手开放平台是否可开通等价评论权限。

### 14.2 待确认问题

1. 服装产品信息来源是什么：人工录入、Excel、ERP、图片文件夹，还是电商店铺。
2. 2026-06-09 演示时是否必须展示三个平台真实账号发布，还是允许重点演示小红书真实链路、抖音/快手演示发布和漏斗管理。
3. 快手开放平台后台是否可申请评论列表和评论回复能力；如可用，需要确认 scope、接口文档、审核周期和账号重新授权要求。
